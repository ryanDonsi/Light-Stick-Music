## 🔍 근본 원인 분석

### 발견된 이상 현상

**1. halfTempoFix가 작동하지 않음**
```
TOMBOY: halfRatio = 0.6426 (>= 0.6 기준값)
장윤정: halfRatio = 0.6185 (>= 0.6 기준값)
→ 하지만 반박자 수정이 적용되지 않음!
```

**2. doubleRatio가 1.0 이상**
```
진미령: doubleRatio = 1.31 (2배속 신호가 원신호보다 강함!)
별 보러: doubleRatio = 1.11 (2배속 신호가 원신호보다 강함!)
→ 자동상관이 2배속 위치에서 더 강하다는 의미
```

**3. halfRatio와 doubleRatio의 역설**
- halfRatio >= 0.6인 곡들이 반박자 수정을 받지 못함
- doubleRatio > 1.0인 곡들이 정확한 감지를 못함

### 세 가지 가능한 원인

#### A. ODF 생성 문제 (ODF가 잘못된 위치에 피크)
- 원신호(74 BPM)가 아닌 2배속(148 BPM)에서 더 강한 피크
- 해결: ODF 필터링 또는 계산 방식 개선

#### B. 로그-정규 분포(Prior) 편향 (가장 가능성 높음)
- 센터: 500ms (120 BPM)
- 느린 곡(74 BPM = 810ms)에 대해 매우 낮은 가중치
- 빠른 곡(146 BPM = 410ms)에 대해 높은 가중치
- 결과: AC는 74 BPM이 높지만, Prior×AC는 146 BPM이 높음

#### C. halfTempoFix 로직 문제
- 기준값 0.60이 너무 높음
- 기준값을 0.50으로 낮추면 해결될 수 있음
- 단, B가 근본 원인이면 임시방편일 수 있음

## 🔧 가능한 해결 방안

### 옵션 1: halfRatio 기준값 조정 (간단)
```kotlin
private const val HALF_TEMPO_RATIO = 0.50f  // 기존: 0.60f
```
- **장점:** 간단하고 빠름
- **단점:** Prior 편향이 근본 원인이면 임시방편에 불과함
- **예상 효과:** TOMBOY, 장윤정 2곡 고쳐질 가능성 있음

### 옵션 2: Prior 분포 조정 (근본적)
```kotlin
private const val PRIOR_CENTER_MS = 600L    // 기존: 500L (100 BPM로 전환)
private const val PRIOR_STD_OCTAVE = 0.35f  // 기존: 0.25f (더 넓히기)
```
- **장점:** 느린 곡과 빠른 곡 모두에 균형있게 대응
- **단점:** 전체 곡에 영향을 미쳐 다른 곡이 악화될 수 있음
- **예상 효과:** 4개 오류곡 모두 해결 가능, 다른 곡 영향 최소화

### 옵션 3: 동적 Prior 적용 (고급)
- ODF의 에너지 분포 분석
- 느린 곡인지 빠른 곡인지 자동 감지
- Prior 센터를 동적으로 조정
- **예상 효과:** 최고 정확도, 구현 복잡

## 📈 신호 품질 분석

### 자동상관 (AC) 통계
```
평균 AC: 0.001293
범위: 0.000122 ~ 0.005308
매우 약함 (<0.0005): 15곡 (35%)
```
- **의미:** 리듬 신호가 약한 곡들이 많음
- **가능 원인:** 신스팝, 느린 발라드, 성글어진 리듬

### 반박자 신호 (Half Ratio) 통계
```
반박자 신호 있는 곡: 43곡 (100%)
평균 halfRatio: 0.7261
높음 (>=0.6): 32곡 (74%)
```
- **좋은 신호:** 반박자 신호가 충분히 감지되고 있음
- **문제:** 감지되고도 사용되지 않음

### 2배속 신호 (Double Ratio) 통계
```
2배속 신호 있는 곡: 29곡 (67%)
평균 doubleRatio: 0.9755
```
- **의미:** 대부분 2배속 신호는 약함
- **그런데:** 오류 곡들은 1.1 이상으로 강함 (이상 신호)

## 📊 알고리즘 흐름

```
1. PCM 오디오 입력
   ↓
2. 멀티밴드 플럭스 ODF 생성 (3개 주파수 대역)
   ↓
3. 자동상관 분석 (각 lag마다 AC 계산)
   - lag=37 (162 BPM), lag=41 (146 BPM), ... lag=100 (60 BPM)
   ↓
4. 로그-정규 Prior 가중치 적용
   - 120 BPM 중심, σ=0.25 옥타브
   - Score = AC × Prior
   ↓
5. 최고 Score를 가진 lag 선택
   - TOMBOY: lag=41 (146 BPM) 선택 ← 문제!
   - 정답: lag=74 (81 BPM)이어야 함
   ↓
6. 반박자 검사 (halfTempoFix)
   - halfRatio = AC[lag/2] / AC[lag]
   - TOMBOY: 0.6426 (>= 0.60) → 수정해야 하는데 미적용!
   ↓
7. 비트 추적 (DP 알고리즘)
   ↓
8. 결과 반환
```

## 🔬 기술 상세 정보

### V1 상수 값
```kotlin
PRIOR_CENTER_MS = 500L      // 120 BPM 선호
PRIOR_STD_OCTAVE = 0.25f    // 가우시안 폭
HALF_TEMPO_RATIO = 0.60f    // 반박자 기준값 ← 문제!
MIN_BEAT_MS = 350L          // 최소 비트 (171 BPM)
MAX_BEAT_MS = 1000L         // 최대 비트 (60 BPM)
```

### Prior 수식
```
logRatio = ln(lagMs / 500ms) / ln(2)
prior = exp(-0.5 × (logRatio / 0.25)²)

예:
- lagMs=410ms (146 BPM): prior ≈ 0.98
- lagMs=810ms (74 BPM): prior ≈ 0.65
→ 느린 곡이 0.65/0.98 ≈ 0.66으로 엄청 불리함!
```
