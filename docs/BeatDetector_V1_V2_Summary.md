# BeatDetector V1 vs V2 - 비교 요약

**기준**: 33곡 멀티스레드 환경 (3개 워커)

---

## BeatDetector V1/V2 비교표

| 항목 | **BeatDetectorV1** | **BeatDetectorV2** |
|------|------------------|------------------|
| **참조 알고리즘** | librosa `beat_track` | madmom 풀 파이프라인 |
| **ODF (특징 추출)** | IIR 3밴드 에너지 | SuperFlux (24밴드 로그 필터뱅크) |
| **BPM 추정** | Autocorrelation + Log-Normal Prior | DBN HMM Forward (Viterbi-max) |
| **Hop 크기** | 10ms | 10ms |
| **하모닉 보정** | half-tempo ratio 체크 (60%) | 910ms guard + combPriorScore |
| **외부 의존성** | 없음 | JTransforms FFT 라이브러리 |

## 주요 차이점 상세

### 1️⃣ ODF (Onset Detection Function)

**V1 - IIR 필터 기반**
```
장점: 빠름, 간단함, 분석 가능
단점: 낮은 주파수 분해능
```

**V2 - SuperFlux (24밴드 스펙트럼)**
```
장점: 타악기 onset 선명함, 높은 분해능
단점: STFT 계산 비용 높음
```

### 2️⃣ BPM 추정

**V1 - Autocorrelation**
```
score[lag] = autocorr[lag] × prior[lag]
- 모든 lag을 연속 스윕
- BPM 범위 제약 없음
- 비표준 BPM(66 BPM) 정확함
```

**V2 - DBN HMM**
```
확률 모델 기반 BPM 추정
- 120BPM 중심 prior (Gaussian)
- 템포 변화 억제 (λ=100)
- 연속성 강조
```

### 3️⃣ 하모닉 보정

**V1**
- `ratio = autocorr[bestLag/2] / autocorr[bestLag]`
- `ratio ≥ 0.60`이면 빠른 템포 선택
- 문제: 140+ BPM 곡에서 오류 가능

**V2**
- 910ms guard: `r=0.5 && resultMs < 910ms`일 때만 보정 (Kill This Love 910ms/130BPM 정상화)
- combPriorScore: Gaussian prior로 후처리
- 더 정교한 보정 로직

## 향후 최적화 방향

### V1 개선
- Hann window 캐싱
- 필터뱅크 최적화
- Log-Normal Prior 파라미터 튜닝
- 목표: 112.8초 → <100초

### V2 개선
- FFT SIMD 최적화
- 스트리밍 버퍼 메모리 풀링
- 목표: 273.5초 → <200초
